## 1. 密码学选择

### 1.1 为什么选择 Ed25519 而不是 RSA/ECDSA？

**决策：** 使用 Ed25519 作为唯一签名算法。

**理由：**

| 考虑因素 | Ed25519 | RSA-2048 | ECDSA-P256 |
|----------|---------|----------|------------|
| 密钥大小 | 32B 公钥 / 64B 私钥 | 256B 公钥 / 512B 私钥 | 64B 公钥 / 32B 私钥 |
| 签名大小 | 64B | 256B | 64B |
| 安全性 | 128-bit | ~112-bit | 128-bit |
| 实现复杂度 | 低 | 中 | 高（侧信道风险） |
| 确定性签名 | 是 | 否（需 padding） | 否（需随机数） |

**具体优势：**

1. **公钥嵌入 URI**：32 字节公钥编码为 43 字符 Base64URL，适合放入 Agent ID
2. **确定性签名**：无随机数，简化验签逻辑和测试
3. **现代标准**：被 NaCl/libsodium、TLS 1.3、WireGuard 等广泛采用
4. **性能**：验签速度快，适合高频消息场景

**参考：** libp2p、Nostr、Signal 都采用类似设计。

### 1.2 为什么不用 X25519（密钥交换）？

**决策：** 当前版本不实现端到端加密（E2EE）。

**理由：**

1. **最小化核心**：E2EE 增加协议复杂度和实现负担
2. **TLS 已覆盖传输安全**：大多数场景 WSS 已足够
3. **非对称性需求**：Agent 间通信常涉及第三方 Relay，E2EE 需额外密钥交换机制

**后续版本计划：** 在路线图中明确列出了端到端加密作为后续功能。

### 1.3 为什么用 Base64URL 而不是 Hex？

**决策：** 公钥编码使用 Base64URL（无填充）。

**理由：**

| 编码方式 | 32 字节编码结果 | 适用场景 |
|----------|-----------------|----------|
| Hex | 64 字符 | 调试友好，但太长 |
| Base64 | 44 字符（含 padding） | 标准，但含 `+/=` 字符 |
| Base64URL | 43 字符 | URL-safe，最紧凑 |

Agent ID 需要嵌入 URL 和配置文件，Base64URL 是最紧凑的 URL-safe 编码。

## 2. 传输协议

### 2.1 为什么选择 WebSocket？

**决策：** 以 WebSocket 作为唯一传输协议。

**理由：**

1. **全双工通信**：Agent 可主动推送，无需轮询
2. **穿越 NAT**：WebSocket 使用 HTTP 握手，易于穿透防火墙
3. **广泛支持**：所有主流语言和平台都有成熟实现
4. **简单性**：相比 HTTP/2、gRPC、WebTransport，WebSocket 概念最少
5. **代理友好**：通过标准 HTTP 端口（80/443），不易被阻止

**为什么不选其他方案：**

| 方案 | 为什么不选 |
|------|------------|
| HTTP Long-polling | 无推送能力，资源消耗大 |
| HTTP/2 | 需要 TLS ALPN，IoT 设备支持差 |
| QUIC/WebTransport | 相对新，生态系统不成熟 |
| Raw TCP | 需要自己实现帧协议 |
| mDNS 直接 | 仅限局域网，无法跨网络 |

### 2.2 为什么心跳间隔是 30 秒？

**决策：** 默认心跳间隔 30 秒，Relay 链路 15 秒。

**理由：**

1. **NAT 超时**：大多数 NAT 设备在 30-60 秒无活动后清理映射
2. **移动网络**：移动网络的 NAT/PAT 超时通常在 30-60 秒
3. **平衡性**：30 秒足够短以保持连接，又不会产生过多流量
4. **可配置**：协议允许 Agent 声明不同的间隔，适配 IoT 场景

**Relay 链路更短的原因：** Relay 作为中间人，需维护大量并发连接，更短的心跳可更快检测断线。

## 3. 身份模型

### 3.1 为什么 Agent ID 直接嵌入公钥？

**决策：** Agent ID 的 user 段为 Base64URL 编码的 Ed25519 公钥。

**理由：**

1. **自认证**：任何人从 ID 即可提取公钥验签，无需查表
2. **无 CA 依赖**：无需证书颁发机构
3. **简洁**：身份与密钥不可分离
4. **可比性**：不同通道获取的 Agent ID 可直接比对

**类比：**
- libp2p peer ID：公钥哈希
- Nostr npub：公钥 Bech32 编码
- SSH 指纹：公钥哈希（但绑定到 hostname）

ADP 直接用公钥而非哈希，提供了更好的可验证性。

### 3.2 为什么引入 visual_code？

**决策：** 从公钥派生 8 字符短码用于人工比对。

**理由：**

1. **43 字符太长**：人工比对不现实
2. **二维码/口述场景**：需要简短的标识符
3. **安全足够**：8 字符 BLAKE2b 哈希，碰撞概率约 2^48

**为什么不直接用短公钥？**
- 短公钥会降低暴力碰撞的难度
- BLAKE2b 提供更好的均匀分布

### 3.3 为什么密钥轮换需要显式声明？

**决策：** 旧身份发送 `adp:key.rotate` 消息传递信任给新身份。

**理由：**

1. **不可伪造**：轮换必须由旧私钥签名，他人无法假冒
2. **可追溯**：trust store 保留历史，支持审计
3. **选择性传递**：仅向收到声明的对端传递，不广播

**为什么不自动处理？**
- 自动处理可能造成身份混淆
- 显式声明给用户确认机会

## 4. 消息格式

### 4.1 为什么用单一消息格式？

**决策：** 不区分 request/response/notification，统一使用 Envelope。

**理由：**

1. **简化**：实现者只需处理一种消息类型
2. **正交性**：方向由 `reply_to` 表示，类型由字段组合决定
3. **扩展性**：新增变体无需新增消息类型

**识别消息语义的方式：**

```
无 reply_to + 无 error  →  新消息
有 reply_to             →  回复
有 error                →  错误回复
```

### 4.2 为什么强制签名？

**决策：** 所有 Agent 间 Envelope 必须附带 Ed25519 签名。

**理由：**

1. **安全基线**：无签名消息直接拒绝，防止伪造
2. **去中心化**：不依赖中心化 CA 验证身份
3. **简化信任模型**：TOFU + 签名验证是去中心化场景的最佳平衡

**为什么不是可选？**
- 可选签名会增加实现复杂度
- 可选意味着需要处理混合场景
- 安全协议应该强制而非可选

### 4.3 为什么定义 ADP Canonical JSON？

**决策：** 定义自己的 JSON 规范化算法。

**理由：**

1. **跨语言一致性**：JSON.stringify 在不同实现间可能有差异
2. **可预测性**：规范化结果可预先计算，用于测试向量
3. **简洁性**：相比 RFC 8785，ADP Canonical JSON 更易实现

**为什么不直接用 RFC 8785 (JSON Canonicalization Scheme)？**
- 过于复杂，包含太多边缘情况
- ADP 场景不需要处理所有 JSON 特性（如 Unicode 归一化）
- 实用性优先

## 5. 任务抽象

### 5.1 为什么任务是可选的？

**决策：** 任务抽象（`adp:task.*`）作为薄层添加到核心消息之上。

**理由：**

1. **最小核心**：协议核心只定义消息投递原语
2. **灵活工作流**：不同场景需要不同的工作流语义
3. **不增加复杂度**：任务仍然是标准 Envelope + 签名

**为什么不内置完整任务系统？**
- 复杂任务系统会偏离"轻量协议"的定位
- 大多数 Agent 可能有自己偏好的任务管理方式

### 5.2 为什么只有 5 种状态？

**决策：** 任务状态机为 PENDING → WORKING → COMPLETED/FAILED/CANCELED。

**理由：**

1. **最小化**：这是满足基本追踪需求的最小状态集
2. **足够表达**：涵盖了创建、执行、完成、失败、取消等核心场景
3. **避免复杂**：不引入 WAITING、PAUSED、SUSPENDED 等额外状态

**为什么不引入更多状态？**
- 更多状态 = 更复杂的实现
- 更复杂的客户端需要处理更多边缘情况

## 6. 发现机制

### 6.1 为什么支持三种发现方式？

**决策：** mDNS（局域网）、静态配置、Registry（跨网络）并存。

**理由：**

1. **场景匹配**：不同场景适合不同的发现方式
2. **无强制依赖**：不要求任何中心化基础设施
3. **渐进增强**：从零配置到中心化服务，用户按需选择

**mDNS 的优势：**
- 零配置，插入即可发现
- 无需 Internet 连接
- 局域网内低延迟

**Registry 的定位：**
- 仅存储身份到路由的映射
- 不存储消息，不参与通信
- 可被完全攻破而不影响身份安全

### 6.2 为什么 Registry 是不被信任的？

**决策：** 明确 Registry 不参与任何信任决策。

**理由：**

1. **安全边界清晰**：用户知道 Registry 被攻破的后果
2. **终端验签**：所有身份最终由终端验签确认
3. **最小权限**：Registry 只做目录服务

## 7. 协议版本管理

### 7.1 为什么 MAJOR=0 表示不稳定？

**决策：** 版本格式 `MAJOR.MINOR`，MAJOR=0 表示 API 不稳定。

**理由：**

遵循语义化版本（SemVer）惯例：
- MAJOR 变更 = 不兼容
- MINOR 变更 = 向前兼容
- PATCH 变更 = 向后兼容

v0.2 表示仍在开发中，API 可能变化。

### 7.2 为什么协议版本同时出现在 Manifest 和 Envelope？

**决策：** Manifest 声明 Agent 最高支持版本，Envelope 声明本消息使用的版本。

**理由：**

1. **版本协商**：Agent 可了解对方能力
2. **发送策略**：发送方选择双方都支持的版本
3. **错误处理**：版本不匹配时返回清晰错误

## 8. 未采纳的方案

### 8.1 没有采用 JSON-LD/语义网技术

**原因：** 过于复杂，不适合轻量协议。Manifest 的 JSON 结构已足够表达能力。

### 8.2 没有采用 Protobuf/FlatBuffers

**原因：** 二进制格式增加调试难度。JSON 是人类可读的，便于开发和调试。

### 8.3 没有采用 DID/VC

**原因：** 完整 DID 规范过于复杂。ADP 的自认证身份已满足需求。

### 8.4 没有采用消息队列模式

**原因：** 会增加协议复杂度和实现负担。at-most-once + 上层可靠性是更灵活的设计。
