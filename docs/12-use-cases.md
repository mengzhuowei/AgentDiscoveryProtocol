# ADP 使用场景

**更新：** 2026-05-15

---

本文档收集 ADP 协议的典型使用场景，帮助理解协议的价值，也为协议演进提供用例依据。

---

## 场景一：家庭智能体网络

**背景：** 家庭成员各有自己的 AI Agent（NAS 上跑一个、笔记本上跑一个、树莓派上跑一个）。

**现状痛点：**
- 每个 Agent 独立运行，互相不知道对方存在
- 家人之间需要"帮我问一下我的 Agent 那个事"这种间接沟通

**ADP 的解法：**

```
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Alice 的  │    │  Bob 的   │    │ 家庭 NAS  │
│  Agent    │◄──►│  Agent   │◄──►│   Agent   │
│ (笔记本)  │    │ (手机)   │    │  (常驻)   │
└──────────┘    └──────────┘    └──────────┘
      │              │               │
      └──────────────┴───────────────┘
                      │
                Registry（NAS 本地）
```

- 家庭自建 Registry（或在 NAS 上跑一个实例）
- Agent 之间互相发现，无需手动配置 IP
- Alice 的 Agent 可以"帮我问 Bob 的 Agent 今晚有没有空"

## 场景二：开发者工具链

**背景：** 开发者使用多个 AI 编码工具（Claude Code、Codex、Cline 等），各自为政。

**现状痛点：**
- Claude Code 写完了代码，不知道要通知 Codex 做 review
- 手动复制粘贴上下文，跨工具协作效率低

**ADP 的解法：**

```
Claude Code  ←───────  ADP  ───────►  Codex
  (写代码)                             (审查代码)
     │                                     │
     │   "帮我 review 这个 PR #42"         │
     │────────────────────────────────────►│
     │                                     │
     │   "已审查，发现 3 个问题"           │
     │◄────────────────────────────────────│
```

- 每个编码工具都接入 ADP
- Claude Code 写完 PR 后直接委派给 Codex 做 code review
- 结果自动返回，不需要开发者手动切换工具

## 场景三：分布式任务执行

**背景：** 团队有多个服务器，每台跑着不同的 Agent，有的擅长数据处理、有的擅长 API 开发、有的擅长运维。

**现状痛点：**
- 任务分配靠人手动指定"这个任务你去跑那个服务器"
- Agent 空闲时没法主动接活

**ADP 的解法：**

```
                        Registry
                           │
          ┌────────────────┼────────────────┐
          │                │                │
     ┌────▼───┐      ┌────▼───┐      ┌────▼───┐
     │ Data    │      │  API   │      │  Ops   │
     │ Agent   │◄────►│ Agent  │◄────►│ Agent  │
     └─────────┘      └────────┘      └────────┘
```

- 任务分发 Agent 搜索可用的执行 Agent（按能力搜索 `search?capability=custom:data.process`）
- 找到后直接委派任务
- 多个同类型 Agent 可实现负载均衡

## 场景四：跨组织协作

**背景：** 两家公司的 Agent 需要有限度的互相通信。

**现状痛点：**
- 不可能让对方的 Agent 直接接入自己的内网
- 需要可控的、可审计的协作通道

**ADP 的解法：**

```
公司 A 内网                公网                 公司 B 内网
┌────────────┐      ┌───────────┐      ┌────────────┐
│ Agent A    │◄────►│   Relay   │◄────►│ Agent B    │
│ (内网)     │      │ (VPS)     │      │ (内网)     │
└────────────┘      └───────────┘      └────────────┘
     │                                        │
 各自的 Registry                       各自的 Registry
```

- 双方通过 Relay 中转，不暴露内网 IP
- 通过 Manifest 控制暴露的能力范围
- Agent A 只能调用 B 的 `custom:data.query`，不能调用 B 的内部运维接口

## 场景五：物联网边缘网络

**背景：** 多个 IoT 设备（摄像头、传感器、智能家居中枢）各自跑着轻量 Agent。

**现状痛点：**
- 每个设备需要单独配置云服务
- 设备之间没法直接通信，全靠云中转
- 局域网内的设备互相不知道对方

**ADP 的解法：**

```
┌─────────┐
│ 网关卡   │ ← Registry (本地)
└────┬────┘
     │
┌────┼────┬────┬────┐
│    │    │    │    │
▼    ▼    ▼    ▼    ▼
传感器 摄像头 门锁 灯光 中枢
```

- 轻量 Agent 在设备上运行，只实现 ping + info.share
- 通过局域网直连（无公网暴露）
- 传感器检测到异常 → 通知摄像头 Agent 录像 → 通知门锁 Agent 锁定

## 场景选择建议

| 场景 | 适合谁 | 复杂度 |
|---|---|---|
| 家庭智能体网络 | 个人/家庭 | ⭐ 低 |
| 开发者工具链 | 开发者 | ⭐⭐ 中 |
| 分布式任务执行 | 团队 | ⭐⭐⭐ 中高 |
| 跨组织协作 | 企业 | ⭐⭐⭐ 高 |
| IoT 边缘网络 | IoT/智能家居 | ⭐⭐ 中 |
